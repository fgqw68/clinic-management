# KadeejaClinic - Professional Clinic Management Telegram Bot

A professional Telegram bot for clinic management with role-based access control, patient tracking, pregnancy registry, and task management.

## Features

### Role-Based Access Control (Database-Driven)
- **Admin View**: Full dashboard with clinic performance analytics
- **Staff View**: Workbench for logging visits and completing assigned tasks
- Secure access control using Supabase `staff_config` table

### Core Functionality
- **Patient Registration**: Complete patient profiles with visit history
- **Visit Logging**: 10-state conversation flow with pregnancy tracking
- **Pregnancy Registry**: Track pregnancies with EDC management
- **Smart EDC Logic**: Registry lookup with Keep/Edit functionality
- **Task Management**: Planned calls (patient tasks) and unplanned tasks (admin tasks)
- **Sunday Shift**: Automatic adjustment of due dates falling on Sunday
- **Visit History**: Notes prepended to maintain chronological order

## Database Schema

### 1. `patients` Table
- **Primary Key**: (patient_name, phone_number)
- **Columns**: patient_name, phone_number, patient_id, last_visit_date (DATE), notes (TEXT)
- **Notes History**: New notes are *prepended* to maintain chronological order

### 2. `visits` Table
- **Primary Key**: (patient_name, phone_number, visit_date)
- **Columns**: patient_name, phone_number, visit_date (DATE, DEFAULT CURRENT_DATE), is_pregnancy (BOOLEAN, DEFAULT FALSE), next_visit_planned_date (DATE), followup_status (TEXT, DEFAULT 'Pending'), remarks (TEXT)
- **Foreign Key**: References patients ON UPDATE CASCADE
- **Note**: `edc_date` and `gravida_status` are stored in `pregnancy_registry` table, not here

### 3. `pregnancy_registry` Table
- **Primary Key**: (patient_name, phone_number)
- **Columns**: patient_name, phone_number, edc_date (DATE), status (TEXT, DEFAULT 'Active')
- **Status Values**: 'Active', 'Dropped', 'Unreachable', 'Delivered'
- **Note**: One active pregnancy per patient (no gravida tracking in this schema)

### 4. `bookings` Table
- **Primary Key**: (patient_name, phone_number)
- **Columns**: patient_name, phone_number, planned_date (DATE)
- **Foreign Key**: References patients ON UPDATE CASCADE

### 5. `patient_tasks` Table
- **Primary Key**: (assignee, patient_name, phone_number, followup_type)
- **Columns**: assignee (TEXT), patient_name, phone_number, followup_type (TEXT), status (TEXT, DEFAULT 'Pending'), due_date (DATE, DEFAULT CURRENT_DATE)
- **Status Values**: 'Pending', 'Completed', 'Unreachable', 'Discontinued'
- **Foreign Key**: References patients ON UPDATE CASCADE
- **Note**: `assignee` is staff_name from staff_config table

### 6. `admin_tasks` Table
- **Primary Key**: id (UUID, generated by Supabase)
- **Columns**: id (UUID), assignee (TEXT, DEFAULT 'Nimisha'), task_message (TEXT), status (TEXT, DEFAULT 'Pending'), due_date (DATE, DEFAULT CURRENT_DATE), created_at (TIMESTAMPTZ, DEFAULT NOW())
- **Status Values**: 'Pending', 'Completed'
- **Note**: Uses a single `task_message` field instead of separate title/description

### 7. `staff_config` Table
- **Primary Key**: telegram_id (BIGINT)
- **Columns**: telegram_id, staff_name (TEXT, NOT NULL), role (TEXT, DEFAULT 'staff'), is_active (BOOLEAN, DEFAULT TRUE)
- **Role Values**: 'admin', 'staff'
- **Purpose**: Database-driven staff configuration (replaces hardcoded Python config)

## Setup Instructions

### 1. Install Dependencies

```bash
pip install -r requirements.txt
```

### 2. Configure Environment Variables

1. Copy `.env.example` to `.env`:
   ```bash
   cp .env.example .env
   ```

2. Edit `.env` and add your credentials:
   ```env
   BOT_TOKEN=your_telegram_bot_token_here
   SUPABASE_URL=your_supabase_url_here
   SUPABASE_KEY=your_supabase_anon_key_here
   ```

### 3. Create Database Tables

Run the SQL in `schema.sql` file in your Supabase SQL Editor.

### 4. Configure Staff Access

Add staff members to the `staff_config` table:

```sql
-- Add Admin (Dr. Fareeda)
INSERT INTO staff_config (telegram_id, staff_name, role, is_active)
VALUES (DR_FAREEDA_TELEGRAM_ID, 'Dr. Fareeda', 'admin', TRUE);

-- Add Staff (Nimisha)
INSERT INTO staff_config (telegram_id, staff_name, role, is_active)
VALUES (NIMISHA_TELEGRAM_ID, 'Nimisha', 'staff', TRUE);
```

**To get Telegram Chat IDs:**
1. Start a chat with [@userinfobot](https://t.me/userinfobot)
2. The bot will return your Chat ID

### 5. Get Telegram Bot Token

1. Start a chat with [@BotFather](https://t.me/botfather) on Telegram
2. Send `/newbot` and follow the prompts
3. Copy the bot token provided

### 6. Run the Bot

```bash
python bot.py
```

## Commands

### `/start`
Shows the role-based welcome message and main menu.

### `/visit`
Starts a 10-state conversation to register a patient visit:

1. **Visit Date**: Enter date (YYYY-MM-DD) or click "Today"
2. **Patient Name**: Enter patient's name
3. **Phone Number**: Enter phone number
4. **Patient ID**: Enter or confirm patient ID
5. **Is Pregnancy?**: Yes/No selection
6. **EDC Check** (if pregnancy):
   - Checks registry for existing EDC
   - If exists: Shows current EDC with [Keep] [Edit] [Cancel] buttons
   - If not exists: Prompts for new EDC input
7. **EDC Confirmation** (if new EDC): Confirm the entered EDC
8. **Notes**: Enter clinical notes for the visit
9. **Next Visit**: Enter next planned date or select "none"
10. **Save**: All data saved to database

**Automatic Actions:**
- Patient notes are *prepended* to history
- Visit record is saved
- Pregnancy registry is updated (if applicable)
- 3-Day Feedback task is created for Nimisha
- Bookings table is updated (if next visit set)

### `/search <term>`
Search patients by name or phone number. Shows:
- Patient details (name, phone, patient ID, last visit)
- Last 3 visit notes with pregnancy status

### `/tasks`
Role-based task views:

**Admin**: Shows summary of Completed vs Pending tasks for all staff with completion rates.

**Staff**: Shows their pending tasks:
- Planned Patient Calls (patient_tasks)
- Unplanned Tasks (admin_tasks)

### `/pregnancy_registry` (Admin only)
Shows all active pregnancies in the registry with:
- Patient name and phone
- EDC date
- Current status

### `/create_task` (Admin only)
Creates an unplanned task (admin task):
1. Task message
2. Assignee (defaults to Nimisha)
3. Due date (optional, with Sunday shift)

### `/cancel`
Cancels the current conversation.

## Smart Features

### EDC Storage Logic

**Registry in pregnancy_registry Table**
- EDC is stored in `pregnancy_registry` table
- Uses (patient_name, phone_number) as primary key
- One active pregnancy per patient
- Registry always has the latest EDC date

**Database Sync on Edit**
- When [Edit] is clicked during EDC_CHECK:
  - Updates pregnancy_registry with new EDC
- Ensures consistency between visit records and registry state

### Sunday Shift Logic

All task due dates (Planned or Unplanned) that fall on Sunday automatically shift to Monday.

**Applied to:**
- 3-Day Feedback tasks (created automatically after visits)
- Unplanned tasks (created by admin)
- Any task with a calculated due date

### Prepend Notes Logic

New patient notes are *prepended* to the notes field:
```
[2026-03-27]: New visit notes here
[2026-03-20]: Previous visit notes
[2026-03-10]: Older visit notes
```

This ensures chronological order when reading the history.

## Access Control

Every command is protected by the `@authorized_only` decorator. Staff configuration is pulled from the `staff_config` table in Supabase. Unauthorized users receive:

```
❌ You are not authorized to use this bot.

Please contact the clinic administrator for access.
```

Admin-only commands (`/pregnancy_registry`, `/create_task`) are additionally gated by role checks from the database.

## Troubleshooting

### Bot not responding
- Check BOT_TOKEN in `.env` file
- Verify bot is running (`python bot.py`)
- Check Supabase credentials

### Access denied errors
- Verify staff record exists in `staff_config` table
- Ensure telegram_id matches your actual Telegram ID
- Check that `is_active` is TRUE
- Verify role assignment ('admin' or 'staff')

### Database errors
- Verify Supabase URL and API key in `.env`
- Ensure tables are created using `schema.sql`
- Check Supabase project status
